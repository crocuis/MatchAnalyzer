# Football Prediction Dashboard 재설계 스펙

작성일: 2026-04-19
상태: 사용자 검토용 초안
대상:
- `apps/web/src/App.tsx`
- `apps/web/src/components/MatchTable.tsx`
- `apps/web/src/components/PredictionCard.tsx`
- `apps/web/src/components/CheckpointTimeline.tsx`
- `apps/web/src/components/PostMatchReviewCard.tsx`

## 1. 목표

현재 대시보드는 텍스트 나열에 가까워 정보 우선순위가 약하고, 운영자가 빠르게 스캔하거나 문제 경기를 먼저 찾아가기 어렵다.

이번 재설계의 목표는 다음 세 가지다.

- 리그별 경기 목록을 더 빠르게 훑을 수 있게 한다.
- `리뷰 필요 경기`를 가장 먼저 눈에 띄게 만든다.
- 경기별 핵심 판단은 모달에서 빠르게, 깊은 분석은 별도 상세 페이지에서 읽게 만든다.

즉, 이 화면은 마케팅 랜딩 페이지가 아니라 **운영용 분석 대시보드**로 다시 설계한다.

## 2. 비포지션

이번 재설계는 UI/정보 구조 작업이다. 다음은 포함하지 않는다.

- 예측 모델 로직 변경
- API 계약 확장
- 새로운 데이터 소스 추가
- 실시간 스트리밍이나 소켓 기반 갱신
- 디자인 시스템 전체 재구축

즉, 현재 데이터 구조를 더 잘 읽히게 재배치하는 것이 범위다.

## 3. 선택한 방향

### 3.1 전체 방향

선택안은 다음과 같다.

- `Calm Editorial Operator`
- `리그 탭 + 2열 카드 목록`
- `리뷰 필요 카드는 좌측 액센트 바 + 배지`
- `카드 클릭 시 분석용 상세 모달`
- `전체 내용은 별도 상세 페이지`

이 방향은 텍스트 나열 문제를 줄이면서도 운영 화면 특유의 밀도와 스캔 속도를 유지한다.

### 3.2 visual thesis

차갑고 정돈된 운영 화면 위에서, 중요한 경기만 조용하지만 분명하게 떠오르는 분석 워크스페이스

### 3.3 content plan

1. 리그 선택과 현재 범위 인지
2. 경기 카드 목록 스캔
3. 선택 경기의 빠른 분석
4. 필요 시 전체 리포트 페이지 진입

### 3.4 interaction thesis

- 리그 탭 전환은 짧고 가볍게 반응한다.
- 카드 hover/active는 절제된 강조만 준다.
- 모달은 작은 경고 팝업이 아니라, 분석용 시트처럼 넓게 열린다.

## 4. 정보 구조

### 4.1 첫 화면

첫 화면에서 가장 먼저 보여야 하는 정보는 `경기 목록과 상태`다.

상단 구조:

- 페이지 제목
- 짧은 보조 설명 한 줄
- 리그 탭
- 현재 리그 기준 `경기 수 / 리뷰 필요 경기 수`

본문 구조:

- 데스크톱: 2열 카드 목록
- 모바일: 1열 카드 목록

### 4.2 카드의 역할

카드는 “이 경기를 지금 봐야 하는가”를 빠르게 판단하게 만드는 단위다.

카드에는 핵심만 넣는다.

- 홈팀 vs 원정팀
- kickoff
- 현재 상태
- 추천 픽
- confidence

카드 안에 과도한 설명, 긴 체크포인트 요약, 긴 리뷰 문장은 넣지 않는다.

### 4.3 상세 모달의 역할

상세 모달은 빠른 분석용이다.

모달에서 보여주는 것:

1. 경기 기본 정보
2. 큰 추천 픽 + 작은 확률 바
3. 리뷰 필요 이유
4. 체크포인트 요약
5. `Open full report`

모달은 단일 컬럼으로 위에서 아래로 읽히게 한다.

### 4.4 전체 상세 페이지의 역할

전체 상세 페이지는 모달보다 더 긴 분석을 담는 화면이다.

페이지 상단 우선순위:

1. 예측 결과 요약
2. 체크포인트 변화
3. 리뷰/오차 원인 분석

즉, 상세 페이지의 첫 화면도 여전히 `예측 결과 요약`이 중심이다.

## 5. 레이아웃 설계

### 5.1 상단 리그 탭

리그 탭은 버튼형이 아니라 **텍스트 탭 + 현재 선택 리그 밑줄 강조**로 간다.

이유:

- 카드 목록이 주인공이라 탭이 너무 강하게 튀면 안 된다.
- 운영 화면의 밀도를 해치지 않으면서 현재 범위를 또렷하게 보여줄 수 있다.

탭 아래 보조 줄:

- `12 matches`
- `3 need review`

이 정도의 운영 정보만 짧게 둔다.

### 5.2 카드 그리드

그리드는 2열이 기본이다.

이유:

- 1열은 읽기 쉽지만 정보량이 부족하다.
- 3열은 카드가 너무 작아져 핵심 판단 밀도가 떨어진다.
- 2열이 현재 요구에 가장 균형이 맞다.

### 5.3 카드 강조 방식

`리뷰 필요 경기`는 다음 방식으로 강조한다.

- 카드 전체 배경색 변경은 하지 않는다.
- 좌측 액센트 바를 둔다.
- `Needs Review` 배지를 함께 둔다.

이유:

- 카드 배경을 바꾸면 텍스트 대비가 흔들린다.
- 긴 목록에서 색면이 쌓이면 가독성이 급격히 떨어진다.
- 좌측 액센트 바 + 배지는 우선순위를 주면서도 읽기를 방해하지 않는다.

### 5.4 상세 모달

모달은 작은 팝업이 아니라 **분석 시트**처럼 보여야 한다.

권장 특성:

- 데스크톱에서는 넓은 중앙 모달
- 모바일에서는 거의 전체화면 시트 또는 바텀시트
- 내부는 단일 컬럼

모달 상단 구성:

- 경기명
- kickoff
- 현재 상태
- 큰 추천 픽
- confidence
- 작은 확률 바

### 5.5 전체 상세 페이지

상세 페이지는 보고서에 가깝게 설계한다.

상단:

- 경기 헤더
- 추천 픽
- confidence
- 숫자 + 막대 그래프 조합의 1X2 확률

중단:

- 체크포인트 변화

하단:

- 리뷰/오차 원인 분석

## 6. 컴포넌트별 책임

### App

- 전체 레이아웃 조립
- 리그 선택 상태
- 선택 경기 상태
- 모달 열기/닫기

### MatchTable

이름과 역할이 어긋난다. 이번 재설계 이후에는 표보다 카드 그리드 역할이 더 강해진다.

권장:

- 내부 책임을 카드 목록으로 전환
- 필요하면 후속 구현에서 이름도 `MatchGrid` 또는 유사한 방향으로 정리

### PredictionCard

- 상세 모달과 상세 페이지 상단에서 재사용 가능한 예측 요약 블록
- 큰 추천 픽 + confidence + 확률 바

### CheckpointTimeline

- 상세 모달에서는 요약형
- 상세 페이지에서는 확장형으로 재사용 가능해야 한다

### PostMatchReviewCard

- 카드 목록에서는 쓰지 않는다
- 상세 모달과 상세 페이지에서만 노출한다

## 7. 상태 설계

### 7.1 기본 상태

- `Scheduled`
- `Prediction Ready`
- `Review Ready`
- `Needs Review`

### 7.2 강조 우선순위

가장 강한 강조는 `Needs Review`에만 준다.

다른 상태는 다음처럼 처리한다.

- 텍스트 라벨
- 작고 조용한 상태 배지
- 필요 시 작은 색 포인트

### 7.3 빈 상태

다음 빈 상태는 각각 구분한다.

- 리그에 경기 없음
- 데이터 로딩 실패
- prediction/review가 아직 없음

빈 상태 문구도 마케팅 문장보다 운영 문구로 쓴다.

예:

- `No matches in this league yet.`
- `Prediction data is not available for this match.`

## 8. 모션 원칙

모션은 과장하지 않고, 계층을 읽히게 만드는 데만 쓴다.

반영할 모션:

- 리그 탭 전환 시 짧은 active underline 이동
- 카드 hover 시 얕은 lift 또는 그림자 변화
- 모달 open/close 시 짧은 opacity + translate

모션은 반드시 빠르고 절제돼야 한다.

하지 않을 것:

- 큰 parallax
- 무거운 배경 애니메이션
- 카드마다 서로 다른 개별 모션

## 9. 시각 원칙

### 9.1 톤

- 배경은 밝고 중립적
- 본문 텍스트는 강한 대비
- 강조 색은 하나만 사용
- 붉은 계열은 `Needs Review` 상태에만 집중 사용

### 9.2 타이포

- 제목과 추천 픽에는 분명한 크기 차를 둔다
- 작은 메타 정보는 크기를 줄이되 대비를 충분히 둔다
- 카드 본문은 2~3줄 안에 스캔 가능해야 한다

### 9.3 카드 사용 원칙

`frontend-skill`의 기본값은 cardless지만, 여기서는 카드가 실제 상호작용 단위이므로 허용한다.

단, 다음은 피한다.

- 카드 안에 카드 또 넣기
- 모든 영역에 두꺼운 테두리 넣기
- 과도한 그림자와 배경색 누적

## 10. 반응형 원칙

모바일:

- 1열 목록
- 모달은 거의 전체 화면
- 탭은 가로 스크롤 가능하게

데스크톱:

- 2열 카드
- 넓은 모달
- 상세 페이지에서 더 긴 정보 열람

## 11. 구현 가이드

이 디자인은 구현 시 다음 순서를 권장한다.

1. 정보 구조와 컴포넌트 책임 재정렬
2. 카드 그리드와 리그 탭 구현
3. 상세 모달 구현
4. 전체 상세 페이지 구현
5. 상태/모션/빈 상태 다듬기

## 12. 수용 기준

다음이 만족되면 이 디자인 구현이 성공한 것으로 본다.

- 첫 화면에서 경기 목록이 가장 먼저 읽힌다.
- 현재 리그와 `리뷰 필요 경기 수`를 바로 이해할 수 있다.
- `Needs Review` 경기가 목록에서 즉시 눈에 띈다.
- 카드 안에는 핵심만 있고, 나머지는 모달/상세 페이지로 분리된다.
- 모달만으로 빠른 판단이 가능하다.
- 상세 페이지 상단은 예측 결과 요약이 가장 먼저 보인다.
- 전체 화면이 텍스트 나열처럼 느껴지지 않는다.

## 13. 남은 리스크

- 현재 컴포넌트 이름이 일부 역할과 어긋난다.
- 실제 데이터 shape가 더 풍부해지면 카드/모달 정보 밀도를 다시 조정해야 할 수 있다.
- 상세 페이지 라우팅 구조는 기존 앱 셸과 맞물려 별도 결정이 필요할 수 있다.
